# Bugfixes 260628-1

- [x] The results web generic search must honour a PARTIAL typed term as a
      "find the whole family" filter. Searching `Hek_R1` offers 2 sample
      options: `Hek_R1` (the typed term) and `7607STDY14643771` (the canonical
      sample it matched). Searching `hek_r` offers ONLY `7607STDY14643771` — it
      never offers `hek_r` itself. A partial term must ALSO offer the typed term
      itself as a pickable filter, AND picking it must return results for EVERY
      sample that matches the term, even though results are registered under
      canonical sample names: e.g. with results registered for samples Hek_R1
      and Hek_R2, searching `hek_r` and picking the `hek_r` option must return
      BOTH Hek_R1 and Hek_R2 results. Design confirmed with the user: resolve
      the partial term via MLWH substring search at query time (not a literal
      contains match), so the filter finds the whole matching family. Same
      treatment for the analogous study-title case.
    - Two server-side parts (no frontend source change needed; the existing
      `getVisibleGenericSuggestions` already hoists UNLABELLED suggestions into
      the typed-term option and keeps skipping LABELLED ones per 260627-5, so
      Part A's output renders correctly). `results/server.go`:
    - PART A (`appendRegisteredSampleSearchSuggestions` /
      `appendRegisteredStudySearchSuggestions` via new
      `appendTypedTermSearchSuggestion`): when the substring search finds >=1
      REGISTERED match for the term in a dimension, also append an UNLABELLED
      `{FieldKey:"sample"|"study", Value: term}` typed-term suggestion, ordered
      before the labelled canonical matches and deduped via the existing `seen`
      map (so the complete-identifier case like `Hek_R1` emits no duplicate).
      Existing `searchSuggestionSubstringMinLength` (3, byte length) and limit
      caps respected. So `hek_r` now offers BOTH `hek_r` (applies `sample=hek_r`)
      and the labelled canonical `7607STDY14643771`.
    - PART B (`handleGetResults` via new `substringExpandSamplePartials` /
      `substringExpandStudyPartials` / `searchStudyPartialIDs` /
      `unresolvedSampleSearchPartials` / `unresolvedStudySearchPartials`): after
      the existing EXACT expansion pipeline, for `sample=`/`study=` filter values
      (>=3 bytes) that the exact path did NOT resolve, additionally run the
      `mlwhSubstringSearcher` (`SearchSamples`/`SearchStudies`, the same resolver
      used by suggestions) and merge the canonical `Sample.Name` /
      `Study.IDStudyLims` into `sampleValues`/`studyValues` (study ids then flow
      through the existing study->sample/run/lane Expand). The downstream
      `instr(...)` contains-match in `results/store.go` then returns results for
      every matching canonical, so `sample=hek_r` returns both Hek_R1 and Hek_R2
      results even though they are registered under canonical names.
    - Exact-id precision: a filter value is treated as "resolved" (and skips the
      substring search entirely) when the exact path produced expansions, so an
      exact id that HAS registered results is unchanged and makes zero
      `SearchSamples`/`SearchStudies` calls (proven by regression). An exact id
      that resolves but has ZERO registered results falls through to one bounded
      substring search as a safe "did-you-mean" fallback (it can only ADD
      related results when the exact filter would otherwise return nothing, and
      it is timeout-bounded + degrades) -- a deliberate, documented edge.
    - Gating uses per-value attribution (a `resolved` set threaded through
      `expandSampleSearchValues`/`expandCandidateSampleSearchValues` +
      `markSampleSearchValueResolved`, and a `studySearchValueCanonical` 4th
      return from `canonicalStudySearchValues`). Substring calls are bounded by
      `context.WithTimeout(ctx, mlwhSuggestionSearchTimeout)` and
      `substringSearchDegrades` (never-synced/not-found/unsupported/deadline/
      cancelled => no expansion, no 502); a genuine upstream error still surfaces
      as `ErrMLWHFailed` (502), mirroring `appendSubstringMLWHSearchSuggestions`.
      Reused existing helpers/constants (`mlwhSubstringSearchCap`,
      `searchSuggestionSubstringMinLength`, `mlwhSuggestionSearchTimeout`,
      `substringSearchDegrades`, `mergeSearchValues`, `nonEmptySearchValues`).
    - Tests: `results/server_test.go` `TestServerSearchSubstringFilterResolution`
      (partial sample expand returns both family results; partial study expand;
      exact supplier id triggers no substring call and does not broaden;
      never-synced + deadline degrade with no 502; genuine upstream 502) and 2
      new `TestServerGetSearchSuggestions` cases (partial offers typed-term +
      labelled canonical with ordering; complete identifier offers typed-term +
      canonical with no duplicate); `mockSearchExpander` gained
      `searchStudiesCalls`/`searchSamplesCalls` recorders. One Vitest regression
      in `frontend/tests/filter-builder.test.ts` (unlabelled typed-term +
      labelled canonical for the same field both render; typed one applies its
      own value). No prior regression test weakened. Preserves 260627-5/-6 and
      260618-5 behaviours. No schema migration; `wa mlwh sync` not run.
    - Gates: `make lint-go` (0 issues), `make test-go` (results + mlwh ok; only
      pre-existing load-dependent `cmd` `TestRunDev*` flakes, which pass in
      isolation and are untouched by this diff), `make lint-frontend`,
      `make test-frontend` (553 + new) all pass.
