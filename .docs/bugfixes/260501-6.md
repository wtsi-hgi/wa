## Bugfix 6: Hierarchical enrichment data structure

### Root cause
The seqmeta enrichment API returned flat arrays (`studies`, `samples`, `libraries`) that lost the hierarchical relationships between entities:
- Study → Libraries → Samples → Lanes

This made it difficult for the UI to display structured relationships, as samples were not grouped by their parent library, and lanes (multiple sequencing runs for the same sample) were not distinguished.

### Regression tests added

#### Go tests (seqmeta/enrich_hierarchy_test.go)
1. **H1**: Study enrichment preserves library-to-sample hierarchy
   - Verifies that `StudyDetail` contains `LibraryDetails`, each with its own samples
   - Proves samples are correctly grouped by library type

2. **H2**: Sample enrichment groups lanes by run_id and lane
   - Verifies that `SampleDetail` contains a `Lanes` array
   - Proves multiple sequencing runs are tracked as separate lane entries

3. **H3**: RunID enrichment groups samples by study and library
   - Verifies that run data returns `StudyDetails` (plural) for cross-study runs
   - Proves hierarchy is maintained when samples come from multiple studies

4. **H4**: Library type enrichment groups samples hierarchically by study and library
   - Verifies library type queries group samples by study and library
   - Proves cross-study library types preserve hierarchy

#### Frontend tests (frontend/tests/contracts.test.ts)
1. **Hierarchical study_detail contract**: Validates the Zod schema for `StudyDetail` with nested `LibraryDetail` and samples
2. **Hierarchical sample_detail contract**: Validates the Zod schema for `SampleDetail` with nested `LaneDetail` array

### Implementation changes

#### Backend (Go)
**seqmeta/types.go**:
- Added `LaneDetail` struct to represent sequencing lanes (id_run, lane, tag_index)
- Added `SampleDetail` struct to group a sample with its lanes
- Added `LibraryDetail` struct to group a library with its samples
- Added `StudyDetail` struct to group a study with its library details
- Extended `EnrichmentGraph` with new hierarchical fields:
  - `StudyDetail *StudyDetail`
  - `StudyDetails []StudyDetail`
  - `SampleDetail *SampleDetail`

**seqmeta/enrich.go**:
- Added `buildStudyDetail()` to construct hierarchical study structure from flat samples
- Added `buildStudyDetails()` to construct multiple hierarchical studies
- Added `buildSampleDetail()` to group lanes for a sample
- Updated `enrichStudy()` to populate `StudyDetail`
- Updated `classifySampleIdentifier()` to populate `SampleDetail`
- Updated `enrichStudiesForSamples()` to populate `StudyDetails`

#### Frontend (TypeScript)
**frontend/lib/contracts.ts**:
- Added `laneDetailSchema` and `LaneDetail` type
- Added `sampleDetailSchema` and `SampleDetail` type
- Added `libraryDetailSchema` and `LibraryDetail` type
- Added `studyDetailSchema` and `StudyDetail` type
- Extended `enrichmentGraphSchema` with hierarchical fields

### Backward compatibility
All existing flat fields (`studies`, `samples`, `libraries`) are preserved and continue to work. The new hierarchical fields are **additive** and optional, ensuring no breaking changes for existing consumers.
