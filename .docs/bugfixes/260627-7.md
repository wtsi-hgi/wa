# Bugfixes 260627-7

- [x] A result set registered with sample `7607STDY14643771` (supplier_name
      `Hek_R1`) is not offered as a Sample suggestion when typing `Hek` in the
      results web search box; you have to type the full `Hek_R1`. Root cause
      (confirmed against the real MySQL cache): the substring-suggestion path
      (`appendRegisteredSampleSearchSuggestions` in `results/server.go`) calls
      `SearchSamples(term, mlwhSubstringSearchCap=50, 0)` and only intersects
      those first 50 matches with registered results. `Hek` matches 285 samples
      and `7607STDY14643771` is ~#78 by `id_sample_tmp`, so it falls outside the
      50-row window and is never checked for registration. Typing the full
      `Hek_R1` works because that goes through the exact `ClassifyIdentifier`
      path, which is targeted and uncapped. The cap of 50 was an arbitrary bound
      with no good justification. Fix: raise `mlwhSubstringSearchCap` to the
      mlwh server's maximum search page size (1000; the server 400s a larger
      limit, so 1000 is the effective ceiling), for both the study and sample
      substring-suggestion scans. Measured latency on the real MySQL cache is
      ~0.07s for `Hek` and ~0.37s for a 10000+-match prefix at limit 1000, so it
      stays web-responsive. Residual bound (documented): for a prefix matching
      more than 1000 samples, a registered match beyond the first 1000 by id
      order still won't surface as a suggestion until the prefix is narrowed —
      this is the mlwh server's hard search-page limit, not a new cap.
    - `mlwh/server.go`: exported `SearchMaxLimit = 1000` (the search endpoints'
      max page) and made the internal `mlwhSearchMaxLimit` its alias — one
      source of truth, no behaviour/contract change.
    - `results/server.go`: `mlwhSubstringSearchCap` 50 → `mlwh.SearchMaxLimit`,
      so both the study and sample substring-suggestion scans over-fetch up to
      the server max before intersecting with registered results.
    - Test (`results/server_test.go`): regression in `TestServerGetSearchSuggestions`
      — a registered sample at index 150 of 200 matches is suggested; verified
      to FAIL at cap 50 (window never reaches it) and pass at the raised cap.
    - Verified end-to-end against the real MySQL cache: `search Hek` now yields
      the `7607STDY14643771` Sample suggestion (cap 50 did not); ~0.07s for the
      285-match prefix, ~0.37s for a 10000+-match prefix at the cap.
    - Gates: `make lint-go` (0 issues) and `make test-go` (all packages ok).
