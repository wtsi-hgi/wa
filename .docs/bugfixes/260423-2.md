- [x] No buttons in the results UI work when clicked, except for links to latest result sets and “Download file”. So can’t add filters or do something with columns or collapse/expand file browser tree or open file preview thumbnails to larger size.
	- Root cause: Next.js 16 blocks cross-origin requests to `/_next/*` dev resources. When the dev server is reached via an internal Sanger hostname (e.g. `farm22-wrstat01.internal.sanger.ac.uk`) the JS chunks never load, React never hydrates, and every onClick handler is dead — only plain `<a>` navigation (result links, Download file) still works. Evidence: “Blocked cross-origin request to Next.js dev resource /_next/webpack-hmr” warning in `logs/frontend.log`. Prior agents could not reproduce because Playwright and curl hit `localhost` where the check is satisfied.
	- Fix: `frontend/next.config.ts` now computes `allowedDevOrigins` from a default list (`localhost`, `127.0.0.1`, `*.sanger.ac.uk`, `*.internal.sanger.ac.uk`) merged with comma-separated `WA_DEV_ALLOWED_ORIGINS`, gated behind `NODE_ENV !== "production"`. `run-dev.sh` auto-populates that env var with the current `hostname -f`/`hostname -s` plus any caller-supplied additions before launching `pnpm dev`.
	- Regression coverage: `frontend/tests/next-config.test.ts` exercises `resolveAllowedDevOrigins()` for defaults, env merging, trimming and dedup; new Convey scenarios in `cmd/run_dev_test.go` assert `run-dev.sh` exports the hostnames and appends caller-supplied values.

- [x] seqmeta_sampleid not showing additional info in the individul result view (“enrichment unavailable” for SANG205). Not sure if this is a failure of the feature or of the test setup when I have a real SAGA_API_TOKEN.
	- Root cause: `seqmeta.Validate` speculatively calls `SAGAProvider.GetStudy(identifier)` first. If the real SAGA MLWH backend rejects a non-study-shaped ID (e.g. `SANG205`) with any non-404 status, the old code returned that error immediately, so `/validate/SANG205` became 502 and the frontend showed "enrichment unavailable" even though SANG205 resolves cleanly as a sanger sample ID via `AllSamples`.
	- Fix: `seqmeta/validate.go` now treats any non-nil `GetStudy` error as "not a study" and falls through to the samples/projects lookups. Real SAGA outages still surface because the subsequent `AllStudies`/`AllSamples`/`ListProjects` calls propagate their errors unchanged.
	- Regression coverage: two new Convey scenarios in `seqmeta/validate_test.go` assert that a 422 `saga.APIError` from `GetStudy` (a) falls through to identify `SANG205` as `IdentifierSangerSampleID` and (b) surfaces `ErrUnknownIdentifier` (→ 404) when no sample/project matches.
