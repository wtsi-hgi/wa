# Bugfixes 260627-9

- [x] A multi-word sample search of two unrelated COMMON words (e.g.
      `wa mlwh search "homo sapiens"`) takes ~33s: each token matches 20000+
      samples and the AND scans the union of both index ranges (~6.21M token
      rows, verified by EXPLAIN) via a temp-table GROUP BY that cannot stop at
      the page limit. This is the inherent "intersect two large posting lists"
      cost of the word-token index (FTS, the usual fast-AND tool, was tried and
      rejected at this scale in 260627-1); the nested case `mus musculus` is
      already handled by 260627-8. Decision (user): keep the algorithm (it
      returns correct results; such broad two-common-word terms match a huge set
      anyway) but add a safety timeout so a slow search cannot tie up the hot
      paths. Fix: (1) the results web search-suggestions path
      (`appendSubstringMLWHSearchSuggestions` in `results/server.go`) wraps its
      MLWH substring search (`SearchStudies`/`SearchSamples`) in a bounded
      context deadline; on deadline/cancellation it degrades to "no MLWH
      substring suggestion" (like the existing never-synced degrade) so the
      endpoint still returns its store + exact suggestions promptly instead of
      blocking — it must NOT turn a slow term into a 502. (2) `wa mlwh search`
      (`cmd/mlwh_search.go`) surfaces a deadline-exceeded search as a clear
      "search timed out; a term of two common words can match a very large
      number of samples — try a more specific term" message rather than a raw
      error/hang. No change to the search algorithm/results, no mlwh schema or
      endpoint change.
    - `results/server.go`: the substring-suggestion phase wraps its
      `SearchStudies`/`SearchSamples` calls in a 2s deadline
      (`mlwhSuggestionSearchTimeout`); `substringSearchDegrades` now also returns
      true for `context.DeadlineExceeded`/`context.Canceled` (detected via the
      `errors.Is` chain, which survives the RemoteClient's `ErrUpstreamImpaired`
      wrap), so a slow term yields no MLWH suggestion (HTTP 200; store + exact
      suggestions still returned) instead of blocking or 502. A genuine
      non-deadline upstream failure still surfaces as before.
    - `cmd/mlwh_search.go`: a 15s `mlwhSearchTimeout` bounds the invocation
      (also covers local-cache mode, which has no client cap); a
      deadline-exceeded search prints "search timed out: ... try a more specific
      term" (`isSearchTimeoutError`), no raw error and no `wa mlwh sync` hint.
    - Tests: degrade-on-timeout suggestion case (blocking fake searcher → 200,
      MLWH sample omitted, store suggestion kept; verified to 502 before the
      predicate change) and the CLI timeout-message case; existing
      suggestion/search tests stay green (fast fakes finish under the timeouts).
    - Verified end-to-end against the real MySQL cache: the `homo sapiens`
      suggestions degrade in ~2.1s (HTTP 200, no hang); the CLI search for that
      term prints the message at 15s instead of ~33s; `musculus` and
      `mus musculus` are unaffected (~0.06-0.07s).
    - Gates: `make lint-go` (0 issues) and `make test-go` (all packages ok).
