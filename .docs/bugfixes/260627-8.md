# Bugfixes 260627-8

- [x] `wa mlwh search "mus musculus"` takes ~3.5s while `mus` or `musculus`
      alone are ~0.07s (both 10000+ matches). Root cause: the multi-word sample
      search (260627-6) ANDs the query tokens via
      `GROUP BY id_sample_tmp HAVING MAX(CASE WHEN <range_i> ...)=1` over the
      UNION of the per-token index ranges, which is O(sum of per-token range
      sizes) and cannot stop at the page limit. For `mus musculus` the tokens
      are `["mus","musculus"]` and `mus` is a prefix of `musculus`, so the union
      is dominated by the huge redundant `mus*` range even though the result is
      exactly "samples with a `musculus*` word". Fix (sample search only; study
      search is a plain substring `LIKE '%term%'` and is unaffected): in the
      query-token preparation, drop any query token that is a prefix of another
      retained query token. This is AND-correctness-preserving: any word matching
      the longer prefix also matches the shorter, so requiring both is equivalent
      to requiring just the longer one, and the result set is unchanged.
      `mus musculus` then collapses to the single token `musculus`, takes the
      fast single-range path (~0.07s), and returns the same samples as searching
      `musculus`. Out of scope here: two UNRELATED common words (e.g.
      `homo sapiens`, ~33s) are a genuine intersection of two large posting sets
      that this word-token index cannot do quickly without a different index
      design; that remains a documented limitation, not fixed in this item.
    - `mlwh/search.go`: added `dropPrefixSubsumedTokens`, applied in
      `sampleSearchQueryBounds` BEFORE the single-vs-multi-token branch, so a
      term whose tokens nest (e.g. `mus musculus`) collapses to the most-specific
      token (`musculus`) and takes the fast single-range index seek. Drops any
      query token that is a proper prefix of another retained token; unrelated
      tokens (`homo sapiens`, `hek r1`) are all kept so their AND still applies.
    - Tests (`mlwh/search_test.go`): `SearchSamples("mus musculus")` returns the
      same set as `SearchSamples("musculus")` and excludes a `Mus spretus`
      sample (mus-word but no musculus-word); EXPLAIN-plan assertion that the
      collapsed term uses the single-range plan with no GROUP BY; helper unit
      tests (chains, order, unrelated kept); existing `hek r1` AND test green.
    - Verified end-to-end against the real MySQL cache: `mus musculus`
      ~3.49s → ~0.087s (matches `musculus` alone), 20 returned rows + count
      byte-identical to `musculus`; `hek r1` still ANDs (16 samples, unchanged);
      `homo sapiens` unchanged (~28s, out of scope).
    - Gates: `make lint-go` (0 issues) and `make test-go` (all packages ok).
