# Bugfixes 260627-6

- [x] The search seems broken, at least via the CLI: `wa mlwh search Hek_R1`
      returns `Studies (0):` / `Samples (0):`, yet `wa mlwh info Hek_R1` resolves
      the sample (supplier_name `Hek_R1`, name `7607STDY14643771`), and
      `wa mlwh search Hek` returns 2 studies + 285 samples (including HEK
      samples). So a more specific term that is the literal supplier_name finds
      nothing while a shorter one finds plenty.
    - Root cause (samples only): `SearchSamples`/`CountSampleSearch` match a
      SINGLE word-token prefix against the `sample_search_token` index, but the
      tokenizer (`sampleSearchTokens`) splits on non-`[a-z0-9]`, so `Hek_R1`
      becomes the two tokens `hek` + `r1`, not a single token prefix. The
      260627-5 `sampleTokenPrefixMatchable` short-circuit also returned empty
      immediately for any term with a byte outside `[a-z0-9]` (here the `_`), so
      `Hek_R1` never reached a query. The study side is unaffected: study search
      is a case-insensitive substring (`LIKE '%term%'`), so `Hek` matched the
      `hek` inside `neurothekeoma`; 0 studies for `Hek_R1` is correct.
    - `mlwh/search.go`: `SearchSamples`/`CountSampleSearch` now tokenise the term
      with `sampleSearchTokens` (the same tokeniser used for stored values) and
      match samples that have a word-prefix for EVERY query token. Zero tokens
      give an empty result; one token keeps the unchanged fast single-range
      path; N>1 tokens use an index-backed OR-union with a per-token
      `GROUP BY id_sample_tmp HAVING MAX(CASE ...)=1` (the per-token MAX form is
      correct even when token prefix ranges overlap). Removed the over-aggressive
      `sampleTokenPrefixMatchable` short-circuit and refreshed the
      bounds/successor doc comments. Fixes both `wa mlwh search` and the results
      web suggestions (both call `SearchSamples`); study substring search is
      untouched.
    - Invalid-UTF-8 guarantee preserved: query tokens are always `[a-z0-9]`, so
      `bytePrefixSuccessor` never sees a non-ASCII byte (the resolved PR #23
      Copilot concern stays addressed). A non-ASCII term like `cafe-accent` now
      usefully searches its ASCII token, while an all-non-token term returns
      empty with no bad bound. Minimum term length 3, word-prefix semantics
      ("mus" still matches "Mus Musculus"), and the exact-to-10000 count floor
      are preserved, and `CountSampleSearch` equals the full `SearchSamples`
      length for multi-word terms.
    - Tests (`mlwh/search_test.go`, `mlwh/search_mysql_test.go`): a failing repro
      (`Hek_R1` now found), AND-requires-all-tokens with decoys, overlapping-token
      correctness, single-word unchanged, non-ASCII handling, count equals
      search, min length 3, and the count cap; dual-dialect sqlmock SQL updated;
      removed the obsolete matchable-helper unit test and a stale contradictory
      non-ASCII-returns-empty assertion.
    - Perf (throwaway 1M-sample synthetic cache): a selective multi-word term
      like `hek r1` was about 29ms search / 56ms count; single-token unchanged
      (about 1ms). Documented limitation (not a blocker): two ultra-common words
      are O(union size), under 1s at 1M, and the web-suggestions path is exposed
      too; a future rarest-token anchor or union cap could bound it.
    - Gates: `make lint-go` (0 issues) and `make test-go` (all packages ok).
