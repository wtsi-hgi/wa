# Bugfixes 260627-10

- [x] Searching `hek_r` in the results web UI gives no Sample suggestion and the
      CLI takes ~11s; `homo sapiens` takes ~33s. Cause: a partial/multi-word term
      tokenises to multiple word-tokens and one is a very broad prefix (`r`,
      `homo`, `sapiens`), so the multi-token word-AND (260627-6) scans the union
      of both index ranges and is slow, and the 260627-9 suggestion timeout then
      degrades it to no suggestion. Decision (user): combine the word-token
      search with a fast full-value prefix search so both `hek_r` (a prefix of
      supplier_name `Hek_R1`) and `homo sapiens` (a prefix of common_name) are
      fast, keep the word-token flexibility (separator-insensitive, so `hek r1`
      still finds `Hek_R1`), and remove the slow combine path. A one-time ~10min
      re-sync from the schema bump is user-confirmed acceptable. Targets:
      `hek_r`, `homo sapiens`, `hek r1`, `mus`, `musculus`, `mus musculus` all
      fast and correct; the 260627-9 timeout stays only as a backstop.
    - Part B (schema): `common_name` and `donor_id` are now NOCASE on SQLite / ci
      on MySQL, each with a new `sample_mirror_common_name_idx`, in
      `mlwh/cache_schema/{sqlite,mysql}/sample_mirror.sql`. `CacheSchemaVersion`
      went 6 to 7 and the `run-dev.sh` seed version was bumped 6 to 7 in lockstep
      (a mismatch would let the v7 binary migrate and wipe the seeded test
      cache). The version bump drops and recreates the mirror tables on open and
      re-syncs. Updated the schema-parity, version-pinned, repair, and
      sync-index tests.
    - Part A (full-value prefix): `SearchSamples` and `CountSampleSearch` also
      match samples whose name, supplier_name, common_name, or donor_id has the
      term as a literal case-insensitive prefix, via the NOCASE indexes. The
      helper escapes the LIKE wildcards and keeps literal separators (so `hek_r`
      becomes the escaped prefix pattern), which a NOCASE/ci index can range-seek
      (unlike the BINARY token table). It is gated to multi-token terms, since a
      single token's word-prefix already supersets its full-value prefix.
    - Part C (anchor): the multi-token word-AND now anchors on the most-selective
      searchable (>=3 char) token (bounded count), fetches its samples (bounded
      by the cap), and keeps those whose other tokens (including short ones such
      as `r`) are word-prefixes of some field token, checked in memory. When
      every token is broad it skips the word-AND and lets Part A carry the term.
      The single-token path is unchanged and 260627-8 prefix-subsumption still
      runs first. The slow GROUP-BY helpers were removed.
    - Part D (union): `SearchSamples` returns the de-duped bottom-N union of the
      full-prefix and word-token ids, ordered by id and paged; `CountSampleSearch`
      counts the same union (the cap is reported as a floor). Min term length 3
      and the invalid-UTF-8 guarantee are preserved; study search is untouched.
    - Verified: `make lint-go` (0 issues) and `make test-go` (all packages ok).
      End-to-end on a local SQLite v7 cache: `hek_r` 5ms, `homo sapiens` 26ms,
      `hek r1` 2ms, and `mus`/`musculus`/`mus musculus` under 1ms. On the real
      9.6M-row MySQL cache, via a temporarily-added-then-dropped common_name
      index: `homo sapiens` 48ms and `hek_r` 40ms; the shared cache was then
      restored exactly (still v6, 9.6M rows, 8 indexes) and was never opened with
      the v7 binary.
    - Known non-target behaviour: an all-short-token multi-word term (such as
      `r1 r2`, every token under 3 chars, which was a slow/timed-out word-AND
      before) now matches only via the full-value prefix, because the word-AND
      requires at least one token of 3+ chars.
