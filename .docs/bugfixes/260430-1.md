- [x] Visiting a result set page from `make run` makes the browser non-responsive. The page can get stuck on “Compiling…” or apparently not doing anything, but offering no interaction until Chrome pops up a warning offering to exit the page.
    - Root cause: `SeqmetaCache.set()` invoked the persistence listener synchronously on every set, and `primeSeqmetaCacheEntry` calls `set` many times per enrichment (one per alias: identifier, study_id, study_accession, sanger_sample_id, sample_lims, library_type, plus all `samples`/`libraries` entries). The listener serialized the entire growing cache (`JSON.stringify` + `encodeURIComponent`) and wrote `document.cookie` each time → O(N²) on top of large nested EnrichmentResult graphs. A V8 CPU profile of a cold visit showed `serializeSeqmetaCacheCookie` self time ~115 s and `persistSeqmetaCache` ~66 s, with one PerformanceObserver `longtask` of 58.8 s.
    - Files touched: `frontend/lib/seqmeta-cache-core.ts` (added `pendingFlush` and `scheduleFlush`; `set` now updates the Map synchronously but defers `onChange` via `queueMicrotask`, coalescing many synchronous sets into a single listener call), `frontend/tests/seqmeta-cache-coalesce.test.ts` (regression: 4 cases — coalesced firing, synchronous visibility via get/has/snapshot, equality short-circuit, independent flush windows).
    - Why prior attempts (260429-2, 260429-4) didn't help: they addressed SSR shell delay (loading.tsx, dashboard fetch) but never the client-side cookie-write storm during hydration.
