# Bugfixes 260629-10

- [x] `wa mlwh info` pretty output is not very readable or end-user friendly. Use nice colouring (using golang.org/x/term to detect this is a terminal) and layout and more understandable presentation of info like milestones (eg. instead of listing out time stamps, show it as a progress bar with parts relative to time spent in each phase, still showing compact dates on each transition). Generally come up with nicer, compact, more easy to understand ways of presenting the info, avoiding duplication and semi-repeats of the info due to showing results from each endpoint separately.
  - Replaced the flat key/value-per-endpoint text renderer with a compact, coloured, single-panel-per-entity layout. JSON (`--json`) output, data-fetch (`buildInfoReport`/`populate*`), and the error/sync-hint logic (`infoQueryError`/`classifyForInfo`/`canSync`/`--since`/`--help`) are unchanged; only the human-readable text path changed. Verified `--json` byte-for-byte identical (sample/study/run) before/after.
  - Files: `cmd/mlwh_info.go` (renderer rewritten; `runMLWHInfo` threads an `infoStyle`), new `cmd/mlwh_info_style.go` (colour+width detection) and `cmd/mlwh_info_render.go` (proportional-timeline renderer + pure helpers), tests updated/added in `cmd/mlwh_info_test.go` and new `cmd/mlwh_info_render_test.go`. `go.mod`/`go.sum`: promoted `golang.org/x/term` indirect→direct (now imported directly) and dropped unused `github.com/go-chi/chi/v5` (only a string literal in a test).
  - Colour: enabled only when stdout is a TTY (`golang.org/x/term.IsTerminal`), with `NO_COLOR` (off) and `CLICOLOR_FORCE` (on) overrides; non-tty uses a fixed fallback width (100) so tests/pipes are deterministic and emit no ANSI.
  - Milestones (and run-status events) render as ONE proportional bar: segment widths come from the fixed inter-timestamp (`reached_at`/`entered_at`) deltas — deterministic, never `time.Now()`-based — each segment labelled with its duration, a full-name phase legend above, compact transition dates beneath, and a trailing `▸ current` marker for the open phase. Each closed segment has a min width so its date fits.
  - De-duplication across endpoints: the study panel folds the old separate overview / status-breakdown / samples-with-data / samples-without-data sections into one panel (sample partition shown once as counts + proportional bars sourced from `StatusBreakdown.Distinct`, reconciled with the overview; a single `Cache synced` line; list headers show "N of TOTAL" instead of a misleading capped count). The sample view merges the within-sequencing status (`/sample/:id/progress` runs) and the physical lanes (`/sample/:id/lanes`) into one `Sequencing` section keyed by `id_run`, so a run that appeared twice now appears once (single lane inline, multiple lanes indented under the run).
  - Colour readability (per user feedback): dimming is reserved for genuinely-secondary/decorative content (rule, separators, `(kind)`, UUIDs, `library_id=`/`id_library_lims=` prefixes, parenthetical accession, `Cache synced`, supplementary `by platform`/`library types`/`detailed timeline` lines); important content is rendered at full readability — the milestone transition-date track, `delivered <date>`, and `sequencing <range>` dates. Warnings are yellow (stand out). `uuid_study_lims` is shown on the study panel, dimmed.
  - Deviations (sound, reviewer-confirmed): phase names render as a legend line above the bar (not over each narrow segment); `sanger_sample_id` is hidden when equal to the sample name; cross-year sequencing date ranges include the year; thousands separators added in text only (never JSON).
  - Reviewed PASS (clean-context reviewer): gates pass, JSON contract/error-sync behaviour/data-completeness (`supplier_name`, per-library `library_id`+`id_library_lims`, every iRODS path, both library/study pairings) all preserved (locks in 260512-1 / 260605-1 / 260627-5 / 260623-2). Final gates: `make test-go` (exit 0, all packages) and `make lint-go` (0 issues).
